# DermaCloud

Dermatology clinic management system (HMS) in production at dermacloud.in.

## Stack

- **Framework**: Next.js 15 (App Router, TypeScript, Tailwind CSS)
- **Database**: MongoDB with Mongoose ODM
- **Auth**: JWT-based (middleware.ts handles CORS + auth)
- **Testing**: Vitest — `npm run test:unit`, `npm run test:security`
- **Hosting**: Vercel
- **Storage**: AWS S3 (patient images/files)
- **AI**: Anthropic Claude SDK, Google Generative AI

## Project Structure

```
app/
  api/           → API routes (auth, tier2, upload, contact)
  auth/          → Auth pages (login, signup, forgot/reset password)
  frontdesk/     → Front desk staff module (appointments, patients)
  clinic/        → Doctor module (consultations, pharmacy, billing)
  dashboard/     → Admin dashboard
components/      → Shared UI components
models/          → Mongoose schemas (Patient, Appointment, Consultation*, etc.)
lib/             → Utilities (auth, db, ai, aws, email, sanitize, rate-limit)
tests/
  unit/          → Unit tests (mirrors source structure)
  security/      → Security tests (XSS, injection, auth)
docs/specs/      → Feature/bug spec documents (generated by /pm)
scripts/         → DB seeds, test scripts
```

## Key Patterns

- API routes use `lib/db.ts` for MongoDB connection
- Auth tokens validated via `lib/auth/` utilities
- All user input sanitized through `lib/sanitize.ts`
- Rate limiting via `lib/rate-limit.ts`
- Audit logging via `lib/audit.ts`
- Form configs are dynamic per clinic via `models/FormSettings.ts`

## Conventions

- API routes return `{ success: boolean, data?: any, error?: string }`
- Use Mongoose models from `models/` — don't write raw MongoDB queries
- Tailwind for styling — no CSS modules or styled-components
- Test files mirror source path: `app/api/auth/` → `tests/unit/api/auth/`
- Commit messages reference Jira ticket IDs when applicable

## Commands

- `npm run dev` — start dev server (Turbopack)
- `npm run test` — run all tests
- `npm run test:unit` — unit tests only
- `npm run test:security` — security tests only
- `npx tsc --noEmit` — type check

## Development Workflow

1. `/pm <description>` — create Jira ticket + spec document
2. `/dev <TICKET-ID>` — plan, implement, test, push, open draft PR
3. `/qa <TICKET-ID>` — run tests, validate acceptance criteria
4. `/security-review` — security audit of changes
5. `/review` — code review before merge

## Rules

- Never hardcode secrets — use environment variables
- Never skip auth checks on API routes
- Always sanitize user input before database operations
- Don't add npm dependencies without discussion
- Patient data is sensitive — treat it as PII everywhere
